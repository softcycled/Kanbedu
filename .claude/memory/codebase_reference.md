## Tech stack
- **Framework:** Next.js App Router, React 18, TypeScript strict
- **DB:** Neon (PostgreSQL 16, AWS ap-southeast-1) + Prisma 5. Schema at `/prisma/schema.prisma`. `DATABASE_URL` = pooled, `DIRECT_URL` = direct (for migrations).
- **Auth:** Custom JWT via `jose`. No next-auth. Cookie: `kanbedu-session`, 30-day TTL.
- **UI:** Fully custom components — no shadcn/radix. Tailwind CSS 3 + CSS variable tokens. dnd-kit for drag-and-drop. Recharts for analytics. next-themes for dark mode.
- **Email:** Brevo (REST API in `/src/lib/email.ts`). 300/day free tier.
- **Push notifications:** web-push (VAPID). Subscriptions in `PushSubscription` table.
- **Realtime:** Falls back to 10s polling — Supabase vars removed. `@supabase/supabase-js` still installed but effectively no-op. Needs proper replacement (SSE) eventually.
- **Error tracking:** Sentry (`@sentry/nextjs`).
- **Build:** `prisma generate && prisma migrate deploy && next build`
- **Deploy:** Vercel. Two env files — `.env` (non-secrets) and `.env.local` (local Postgres + secrets).

## Auth architecture
- `getSession()` — validates JWT, checks expiry, checks `passwordChangedAt` (token iat must be after last password change — changing password invalidates all sessions, don't break this)
- `getVerifiedSession()` — same + requires `emailVerified: true`
- `isMemberOfBoard(userId, boardId)` — board access gate
- `getClassRole(userId, classId)` → `"educator" | "ta" | "student" | null`
- Board roles: `"owner" | "member"`. Class roles: `"educator" | "ta" | "student"`.
- Gates before app access: email must be verified + handle must be set (both hard redirects).
- Rate limiting: DB-backed via `RateLimit` table (`/src/lib/rateLimit.ts`).
- **Don't touch:** `KANBEDU_JWT_SECRET` rotation kills all live sessions. Don't modify `passwordChangedAt` logic.

## Code conventions
**API route structure (every route):**
1. `getVerifiedSession()` → 401 if null
2. `const { id } = await params` (Next.js 15+ requires await)
3. `parseBody(schema, raw)` → 400 if invalid
4. Authorization check → 403 if unauthorized
5. Rate limit check for writes
6. DB query with `select` (not `include`)
7. `NextResponse.json(data)` or `{ error: "message" }` with status

**Prisma patterns:** Always `select` specific fields. `Promise.all()` for parallel independent queries. `$transaction()` for multi-step atomicity. Never use `Promise.all` with concurrent awaits on the same `tx` handle inside a transaction — use sequential `for...of` instead.

**Components:** Server components (page.tsx) for auth + initial fetch. `"use client"` for interactive. Heavy modals use `dynamic(() => import(...), { ssr: false })`.

**Naming:** PascalCase components, camelCase hooks/functions, SCREAMING_SNAKE_CASE activity types.

**Path alias:** `@/` = `src/`. Always use this, never relative up-directory paths.

**Fire-and-forget async work** (activities, notifications): `(async () => { ... })()` — doesn't block response.

**dnd-kit `<DndContext>` always needs an explicit `id` prop.** Without it, dnd-kit's internal `aria-describedby` counter is generated by mount order, which can differ between SSR and client hydration when multiple `DndContext` instances exist on a page (e.g. Sidebar's boards/student-classes/educator-classes lists) — causes a silent React hydration mismatch warning on every load. Fixed 2026-06-16; see [[project_changelog]].

**Column color resolution must always go through `resolveColumnPalette(color, index)` from `src/lib/columnPalette.ts`, never `getColumnPalette(index)` alone.** A column's default (unset) color is positional, but an explicit `color` override must win. Also, the `index` must be each component's own consistent ordering — `Board.tsx`/`ColumnHeader.tsx` use `sortedColumns` (done columns always pinned last); passing the raw `columns` array (DB/creation order) to a different component computes a different index for the same column and produces mismatched colors. Bit `TaskModal.tsx`'s Phase dropdown this way (fixed 2026-06-16) — check any new column-color UI against both rules.

**Validation:** All input through Zod schemas in `/src/lib/validations.ts`. Never trust raw request body.

**Markdown syntax (MarkdownText.tsx + MarkdownToolbar.tsx):** `**bold**`, `*italic*`, `***bold+italic***`, `~~strikethrough~~`, `` `code` ``. Underline is `++text++` (changed from `__` on 2026-06-17 — `__dunder__` names were accidentally underlined).
